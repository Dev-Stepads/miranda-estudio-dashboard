1. AUTENTICAÇÃO
--------------------------------------------------------
[OK] Tipo: OAuth 2.0 — fluxo Authorization Code
[OK] Credenciais emitidas pelo Partners Portal:
        client_id        — público
        client_secret    — SECRETO (vai pro .env.local / cofre)

[OK] Fluxo de autorização:
     1. Lojista (Miranda) visita a URL de autorização do nosso app:
        https://www.nuvemshop.com.br/apps/{app_id}/authorize
     2. Lojista aprova → redireciona para nosso redirect_uri com ?code=xxx
     3. Nós trocamos o code por um access_token via POST:

        POST https://www.nuvemshop.com.br/apps/authorize/token

        Headers:
          Content-Type: application/json

        Body (JSON):
          {
            "client_id":     "<client_id>",
            "client_secret": "<client_secret>",
            "grant_type":    "authorization_code",
            "code":          "<código_recebido>"
          }

        Resposta:
          {
            "access_token": "...",
            "token_type":   "bearer",
            "scope":        "read_products,read_orders,...",
            "user_id":      <store_id>    ← precisamos guardar!
          }

[OK] 🟢 Access token NÃO EXPIRA.
     Permanece válido até que o lojista desinstale o app ou
     a gente rotacione manualmente. Isso elimina o pesadelo
     de refresh token que tivemos que planejar pra Conta Azul.

[OK] 🟢 user_id na resposta = store_id da loja Miranda.
     Esse valor vai virar parte da base URL. Precisamos
     persistir no Supabase.

[OK] Scopes necessários (pedir só leitura):
        read_orders            (vendas + carrinhos abandonados)
        read_customers         (clientes, inclui campo extra.gender)
        read_products          (catálogo, inclui variant.sku)
        read_content           (categorias, se quisermos)

     Pedir APENAS escopos de leitura. Nenhum :write.

2. BASE URL E REGRAS GERAIS
--------------------------------------------------------
[OK] Base URL:
       https://api.nuvemshop.com.br/2025-03/{store_id}/{resource}

     2025-03 = versão da API (ano-mês). Vamos travar nessa versão
     até final de 2026 e revisitar. Upgrade costuma ser compatível,
     mas vale planejar.

[OK] Headers obrigatórios em toda request:
       Authentication: bearer {access_token}
         ⚠ ATENÇÃO: é "Authentication" (sic), NÃO "Authorization".
            Essa é uma particularidade da Nuvemshop e é fácil de
            errar se copiar código genérico de OAuth 2.0.
       User-Agent: Miranda Dashboard (dev@stepads.com.br)
         ⚠ OBRIGATÓRIO. Sem User-Agent = 400 Bad Request imediato.
            Formato esperado: "Nome do App (email_ou_url_contato)"
       Content-Type: application/json; charset=utf-8
         Obrigatório só em POST/PUT (sem isso = 415 Unsupported).

[OK] HTTPS obrigatório. Toda request sem TLS é rejeitada.

[OK] Resposta sempre em JSON, snake_case, sem root element wrapper.

[OK] RATE LIMITING — Leaky Bucket algorithm
       Capacidade: 40 requests
       Drenagem:    2 requests/segundo
       Planos Next ou Evolution: 10x (400 capacity, 20 rps)
       Headers de resposta:
         x-rate-limit-limit
         x-rate-limit-remaining
         x-rate-limit-reset
       Exceder → 429 Too Many Requests
       [VER] qual plano a Miranda tem. Se for básico, é 40/2rps.

[OK] PAGINAÇÃO
       Parâmetros:
         page      — 1-indexed
         per_page  — default 30, máximo 200
       Headers de resposta:
         x-total-count   — total de resultados da query
         Link            — URLs com rel=next/prev/first/last
       Limite duro: 10.000 resultados por query.
         → Para coletas grandes, usar filtros de data
           (created_at_min/max) em janelas menores.

[OK] ERROS
       400 — JSON inválido ou User-Agent faltando
       402 — lojista está com pagamento em atraso (Nuvemshop)
       415 — Content-Type faltando em POST/PUT
       422 — campos inválidos
       429 — rate limit
       5xx — servidor. Retry com exponential backoff.

3. WEBHOOKS — 🟢 SUPORTADOS
--------------------------------------------------------
[OK] A Nuvemshop oferece webhooks e isso muda o DESIGN do ETL
     em relação ao que planejamos pro Conta Azul.

[OK] Eventos disponíveis (lista parcial, só os que nos interessam):
       order/created          — pedido novo
       order/updated          — atualização genérica
       order/paid             — pedido pago
       order/cancelled        — pedido cancelado
       order/fulfilled        — pedido entregue
       order/pending          — aguardando pagamento
       order/voided           — estornado
       order/edited           — editado
       checkout/updated       — carrinho mudou
       [VER] checkout/abandoned existe como evento separado?
             Se não existir, usar polling em /checkouts.
       product/created
       product/updated
       product/deleted
       customer/created
       customer/updated
       customer/deleted

[OK] REGISTRO DE WEBHOOK
       POST /2025-03/{store_id}/webhooks
       Body:
         {
           "event": "order/paid",
           "url":   "https://miranda-estudio-dashboard.vercel.app/api/webhooks/nuvemshop"
         }

[OK] PAYLOAD recebido pela Vercel (via POST):
       Headers:
         x-linkedstore-hmac-sha256: <hash para validação>
       Body:
         {
           "store_id": 12345,
           "event":    "order/paid",
           "id":       67890         ← id da entidade (ex: order_id)
         }
       ⚠ O payload NÃO inclui o objeto completo — só o id.
       Depois do webhook, precisamos fazer
       GET /orders/{id} pra buscar os dados atualizados.

[OK] VALIDAÇÃO HMAC — OBRIGATÓRIA
       HMAC-SHA256 do corpo raw da request usando o client_secret
       como chave, comparando com o header x-linkedstore-hmac-sha256.
       Pedidos sem HMAC válido devem ser rejeitados com 401.
       (Sem isso, qualquer um pode enviar POSTs falsos pro nosso
       endpoint.)

[OK] RESPOSTA ESPERADA
       2xx dentro de 3 segundos, caso contrário é considerado falha.
       → Recebimento deve ser ASSÍNCRONO: responde 200 primeiro,
         processa depois (enfileirar em pg_cron ou Supabase Queue).

[OK] RETRY POLICY
       18 tentativas ao longo de 48h:
       imediato, 5min, 10min, 15min, depois exponential backoff.
       Idempotência OBRIGATÓRIA do nosso lado — webhooks podem
       chegar duplicados ou fora de ordem.

[OK] RESTRIÇÃO IMPORTANTE
       URLs do tipo localhost/tiendanube.com/nuvemshop.com.br
       NÃO são aceitas como destino de webhook. Precisa ser
       uma URL pública HTTPS (Vercel resolve isso).

4. ENDPOINTS QUE VAMOS USAR
--------------------------------------------------------

4.1 Listar pedidos                                           [OK]
    GET /2025-03/{store_id}/orders
    Query params mais úteis:
      status          = open | closed | cancelled
      payment_status  = pending | authorized | paid | partially_paid
                        | abandoned | refunded | partially_refunded | voided
      created_at_min  = ISO 8601
      created_at_max  = ISO 8601
      customer_ids    = CSV de ids
      page, per_page  = paginação

4.2 Detalhe de pedido                                        [OK]
    GET /2025-03/{store_id}/orders/{id}
    Já traz os itens (products[]) INLINE — não é endpoint separado
    como no Conta Azul. ✅

4.3 Listar clientes                                          [OK]
    GET /2025-03/{store_id}/customers
    Query params:
      since_id
      created_at_min/max
      updated_at_min/max
      page, per_page, fields
      q     = busca por nome/email/identification
      email = filtro exato por email

4.4 Detalhe de cliente                                       [OK]
    GET /2025-03/{store_id}/customers/{id}
    Inclui o objeto "extra" com gender e outros custom fields.

4.5 Carrinhos abandonados                                    [OK]
    GET /2025-03/{store_id}/checkouts
    Query params:
      since_id
      created_at_max
      updated_at_max
      fields
    ⚠ ATENÇÃO: Retenção de apenas 30 dias para acesso via API,
      90 dias até deleção definitiva. ETL precisa rodar ao menos
      DIÁRIO, preferência a cada 4-6h, para não perder carrinhos
      mais antigos.

4.6 Listar produtos                                          [OK]
    GET /2025-03/{store_id}/products
    Query params relevantes:
      ids (até 30), q, category_id, language
      published, free_shipping, has_promotional_price
      min_stock, max_stock, since_id
      created_at_min/max, updated_at_min/max
      page, per_page, sort_by, fields

4.7 Buscar produto por SKU                                   [OK]
    GET /2025-03/{store_id}/products/sku/{sku}
    ⚠ Útil para o matching Conta Azul ↔ Nuvemshop:
      pegando um produto do Conta Azul (se tiver SKU no
      cadastro lá), consultamos aqui pra casar.

4.8 Webhooks — gestão                                        [OK]
    GET    /2025-03/{store_id}/webhooks       — listar atuais
    POST   /2025-03/{store_id}/webhooks       — registrar novo
    DELETE /2025-03/{store_id}/webhooks/{id}  — remover
    Usaremos no setup inicial (T13) e em caso de precisar
    reinstalar em caso de erro.

6. ENUMS DE STATUS — IMPACTO NO ETL
--------------------------------------------------------

6.1 order.status (ciclo de vida do pedido)

      unshipped
      delivered
      partially_packed

6.4 Regra para ENTRAR no faturamento (a validar com cliente):
      payment_status ∈ {paid}
      [VER] incluir partially_paid? Regra precisa confirmação do

            incluir partially_paid E paid, pois a venda já aconteceu.

8. REGRAS DE NEGÓCIO DESCOBERTAS
--------------------------------------------------------

8.1 Data oficial da venda no canônico:
    Usar paid_at QUANDO payment_status ∈ {paid, partially_paid}
    Fallback para created_at quando paid_at ainda é null
    (ex: pedido pending ou authorized).
    Isso alinha com a decisão (a tomar) sobre "data de venda"
    no REGRAS_CONSOLIDACAO §1.2.
    [VER] bater com representante da Miranda: eles consideram
    a venda "feita" no dia do pedido ou do pagamento? A maioria
    dos relatórios de e-commerce usa paid_at.

8.2 Cliente "Consumidor Final" não existe na Nuvemshop:
    Diferente do Conta Azul, TODO pedido na Nuvemshop tem
    pelo menos contact_email. Checkout exige. Então a aba
    Nuvemshop não terá o problema de "pedidos sem cliente"
    que tivemos na Loja Física.

8.3 Consolidação de produtos por SKU:
    Chave canônica = product.variants[].sku
    Fallback = nome do produto (se SKU ausente dos dois lados).

8.4 Geografia = default_address.province
    Sem fallback necessário (diferente da Loja Física).

8.5 Ticket médio da Nuvemshop:
    total / orders_count onde payment_status IN (paid, partially_paid)
    Respeitando timezone America/Sao_Paulo para agrupamento.

--------------------------------------------------------
9. RISCOS TÉCNICOS
--------------------------------------------------------

R1. Header "Authentication" (sic, não "Authorization")
    É muito fácil copiar código OAuth genérico e escrever
    "Authorization". Vai dar 401 silencioso. Adicionar teste
    unitário no cliente HTTP que assert o header correto.

R2. User-Agent obrigatório
    Qualquer chamada sem User-Agent retorna 400. No boot do
    cliente HTTP, forçar o header padrão.

R3. Versioning 2025-03
    Travado por enquanto. Criar alerta para revisar versões
    a cada 6 meses. Upgrade costuma ser compatível, mas
    queremos controle.

R4. store_id precisa persistir
    Sem o store_id recebido no momento da autorização, não
    conseguimos montar base URL. Guardar no Supabase na mesma
    linha do access_token.

R5. Webhooks duplicados
    Retry de 48h pode entregar o mesmo evento várias vezes.
    Deduplicação por (event_type, resource_id, timestamp) ou
    por idempotency key do lado do ETL.

R6. 30 dias de retenção para abandoned checkouts
    ETL precisa rodar com frequência razoável. Recomendação:
    4x ao dia (a cada 6h) + webhook em tempo real se
    checkout/abandoned existir como evento.

R7. Query de 10k items máximo
    Evitar puxar histórico completo em uma query só. Usar
    janelas de data (created_at_min/max) menores que 30 dias
    para estar seguro.

10. DECISÕES TOMADAS DURANTE O MAPEAMENTO
--------------------------------------------------------

D1. [2026-04-10] [INTEGRACAO] Nuvemshop: webhooks primários +
    polling diário safety-net.
    Contexto: Nuvemshop suporta webhooks com retry de 48h. Polling
    sozinho é desperdício de rate limit. Webhook sozinho pode
    perder eventos se nosso endpoint ficar fora do ar.
    Decisão: webhook é o caminho principal para order/paid,
    order/cancelled, order/updated, customer/created, customer/updated.
    Polling a cada 6h em /checkouts (carrinhos abandonados).
