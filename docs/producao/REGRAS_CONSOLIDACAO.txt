========================================================
REGRAS DE CONSOLIDAÇÃO — Dashboard Miranda Studio
========================================================
Última atualização: 2026-04-10

Este arquivo define como os dados das diferentes fontes
são tratados e unificados. Toda regra aqui deve ser
seguida em todas as transformações e views.

--------------------------------------------------------
1. PERÍODO
--------------------------------------------------------
1.1 Timezone oficial do dashboard: America/Sao_Paulo (BRT/BRST)
    Todo timestamp é convertido para este tz antes de agrupar por dia.

1.2 Data considerada como "data da venda" (decidido em 2026-04-10):
    - Conta Azul: "Data de venda" (campo já visto na UI, nome exato
                   do campo na API a confirmar em T7)
    - Nuvemshop:  paid_at quando payment_status ∈ {paid, partially_paid}
                   FALLBACK para created_at nos casos pending/authorized
                   Quando a venda transita para "paid", o registro
                   canônico é reescrito com a nova paid_at (o ETL
                   trata essa mudança idempotentemente).
    - Meta Ads:   date_start do insight
    Timezone de agrupamento diário: America/Sao_Paulo (§1.1).
    Ver /docs/context/DECISOES.txt 2026-04-10.

1.3 Períodos de comparação padrão no dashboard:
    - hoje
    - últimos 7 dias
    - últimos 30 dias
    - mês atual
    - mês anterior
    - custom (date_from, date_to)

2. STATUS QUE ENTRAM NO FATURAMENTO
--------------------------------------------------------
2.1 Conta Azul — validado em T4 (2026-04-10):
    Status vistos na UI (totalizadores e ficha de venda):
      "Aprovados" (= COMMITTED) → INCLUI no faturamento
      "Previstos" (= PENDING)   → EXCLUI (aguardando confirmação)
      "Cancelados" (= CANCELED) → EXCLUI
    [VER] confirmar se existem status adicionais (Devolvida / Estornada)
          na primeira chamada real à API em T7.
    Filtro adicional: vendedor_responsavel ∈ {valores de "loja física"}
    para limitar ao canal correto. Lista completa de valores válidos
    a ser descoberta na T7/T8.

2.2 Nuvemshop — validado em T5 (2026-04-10):
    Enum completo de payment_status:
      pending, authorized, paid, partially_paid, abandoned,
      refunded, partially_refunded, voided
    INCLUI: payment_status ∈ {paid, partially_paid}
      (partially_paid inclui parcelamentos onde a 1ª parcela caiu)
    EXCLUI: pending, authorized, abandoned, refunded,
            partially_refunded, voided, cancelled
    [VER] confirmar com o representante da Miranda se partially_paid
          deve entrar — nossa recomendação é sim, porque a venda
          já aconteceu. Registrar em DECISOES.txt quando confirmado.

2.3 Faturamento = soma de gross_revenue do que está incluído

--------------------------------------------------------
3. CONSOLIDAÇÃO DE PRODUTOS ENTRE FONTES
--------------------------------------------------------
3.1 Chave primária de matching:
    Ordem de prioridade:
      1. SKU (se existir nas duas fontes)
      2. Nome canônico (normalizado: lowercase, sem acento, trim)
      3. Tabela de equivalência manual em /docs/producao/equivalencias_produtos.csv
         (criar quando necessário)

3.2 Quando duas fontes têm o mesmo SKU com nomes diferentes,
    o canonical_name segue o nome que apareceu primeiro.
    Alterações manuais são possíveis via tabela products.

3.3 Produtos que existem só em uma fonte ainda aparecem
    no ranking consolidado, com revenue da outra fonte = 0.

--------------------------------------------------------
4. DEMOGRAFIA
--------------------------------------------------------
4.0 DISPONIBILIDADE POR FONTE (atualizado 2026-04-10):
    +--------------------+-----------+------------+
    |                    | Gênero    | Idade      |
    +--------------------+-----------+------------+
    | Conta Azul (Loja)  | ❌ NÃO    | ❌ NÃO     |
    | Nuvemshop          | ✅ SIM    | ✅ SIM     |
    +--------------------+-----------+------------+
    Consequência: a aba "Loja Física" do dashboard NÃO tem análises
    por gênero ou faixa etária. Essas análises existem SOMENTE na
    aba Nuvemshop. Ver /docs/context/DECISOES.txt (2026-04-10).

4.1 Gênero:
    Valores canônicos: 'M', 'F', 'unknown'
    Fonte disponível: APENAS Nuvemshop.
    Para source = 'conta_azul': sempre 'unknown' (campo não coletado).
    Fallback geral: 'unknown' quando a fonte não entrega.
    NÃO inferir gênero a partir de nome próprio (viés + falso positivo).

4.2 Idade:
    Faixas: 18-24 / 25-34 / 35-44 / 45-54 / 55+
    Calculada a partir de data_nascimento na data da compra.
    Fonte disponível: APENAS Nuvemshop.
    Para source = 'conta_azul': sempre 'unknown' (campo não coletado).
    Menores de 18 → revisar caso a caso (provavelmente erro de cadastro).
    Sem data_nascimento → 'unknown'.

4.3 Estado (Loja Física — Conta Azul):
    Estado vem DIRETO do endereço estruturado do cliente (cidade + UF)
    na maioria dos casos — validado via screenshot em 2026-04-10.
    Fallback DDD → UF: aplicado APENAS quando o endereço do cliente
    estiver ausente OU quando a venda for para "Consumidor Final -
    Venda de Consumidor" (cliente anônimo do PDV).
    Tabela de DDD → UF deve viver em /src/etl/ddd_to_uf.ts
    DDD inválido ou telefone ausente → state = NULL, inferred = false
    Atenção ao parse do telefone: a Miranda cadastra com código do
    país junto ("(55) 11945-259593"). Parser precisa extrair DDD real
    separando do 55 (país).

4.4 Estado (Nuvemshop):
    Vem direto do shipping_address.province. Sem fallback.

--------------------------------------------------------
5. TICKET MÉDIO
--------------------------------------------------------
ticket_medio = faturamento_liquido / numero_de_pedidos_pagos

Não dividir por itens — é por pedido.
Calculado por fonte e por período independentemente.

--------------------------------------------------------
6. VISÃO GERAL — REGRAS ESPECÍFICAS
--------------------------------------------------------
6.1 Faturamento total = Loja Física + Nuvemshop
    Meta Ads NÃO soma. Faturamento atribuído do Meta é um
    recorte informativo que pode coincidir parcialmente com
    pedidos da Nuvemshop — double-counting proibido.

6.2 Participação por canal:
    share_loja_fisica = revenue_loja_fisica / total_revenue
    share_nuvemshop   = revenue_nuvemshop / total_revenue
    (soma = 100%)

6.3 Ranking consolidado de produtos:
    Soma quantity e revenue por canonical_name, independente
    da fonte. Mantém breakdown por canal.

--------------------------------------------------------
7. META ADS — REGRAS ESPECÍFICAS
--------------------------------------------------------
7.1 ROAS reportado:
    Usar purchase_roas da API quando disponível.
    Fallback: purchase_value / spend.
    Nunca calcular ROAS dividindo por 0 — retornar NULL.

7.2 Atribuição window:
    Decidido em 2026-04-10 (ver DECISOES.txt):
      action_attribution_windows=['7d_click','1d_view']
    Esse é o default atual do Meta (em 2026, depois que 7d_view e
    28d_view foram removidos em 2026-01-12). Os números do dashboard
    vão bater com o Ads Manager da Miranda.
    [VER] validar com o representante da Miranda antes de T25 que
    eles não usam uma janela diferente (ex: só 1d_click).

7.3 Criativos:
    Ranking por ROAS. Em empate, desempate por faturamento absoluto.

--------------------------------------------------------
8. REPROCESSAMENTO
--------------------------------------------------------
A ingestão é incremental mas idempotente.
Reprocessar um período deve chegar ao mesmo resultado.
Raw tables são append-only; canônicas usam UPSERT por
(source, source_id).
