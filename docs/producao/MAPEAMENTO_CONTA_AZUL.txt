## A2. Scopes — CORRIGIDA (Cognito, não granular)

CORRETO:   openid profile aws.cognito.signin.user.admin
ANTIGO:    sales:read, customers:read, products:read  (suposição)

O Conta Azul usa AWS Cognito por baixo (issuer
cognito-idp.sa-east-1.amazonaws.com/sa-east-1_Vp83J11wA,
extraído do JWT). Scopes são FIXOS de identidade Cognito, sem
granularidade por recurso. A autorização efetiva do que o app
pode ler vem do PERFIL DO ERP USER que loga no fluxo OAuth.

Impacto no código: passa esses 3 scopes literais na URL de
authorize. Não tente pedir scopes de recurso — eles não existem
no Cognito pool deles.

## A4. ⚠ PEGADINHA CRÍTICA — /v1/pessoas NÃO é só clientes

O endpoint retorna TUDO que tem cadastro de pessoa no Conta Azul,
incluindo fornecedores, transportadoras, funcionários, clientes.

Campo discriminador: `perfis` (array<string>).

Valores de "perfis" observados:
  - "Fornecedor"      (4 primeiros itens da lista da Miranda são TODOS disso)
  - "Cliente"         (a maioria — precisa filtrar)
  - "Transportadora"  (poucos)
  - "Funcionário"     (poucos)

ETL action: FILTRAR por `"Cliente" in perfis` antes de inserir
na canonical `customers`. Sem esse filtro, vai poluir o canonical
com 4.000+ registros de CNPJs de fornecedores.

## A5. ⚠ Paginação — 3 wrappers diferentes na mesma API

A Conta Azul tem INCONSISTÊNCIA de schema entre endpoints:

  /v1/categorias       → { "itens_totais": N, "itens":  [...] }   (PT-BR)
  /v1/produtos         → { "totalItems":  N, "items":  [...] }    (EN)
  /v1/pessoas          → { "totalItems":  N, "items":  [...] }    (EN)
  /v1/notas-fiscais    → { "itens":  [...] }                      (sem total!)

Impacto no código: o HTTP client do Conta Azul precisa de um
"normalizador de paginação" que aceita os 3 formatos e devolve
uma interface interna única { total, items }. Provavelmente via
um schema Zod com discriminated union, ou um switch no mapper.

Valores aceitos pelo param `tamanho_pagina` (para os 3 endpoints
paginados) são um ENUM fechado:
  10, 20, 50, 100, 200, 500, 1000
Qualquer outro valor → 400 "O tamanho da página deve ser um dos
seguintes valores...". Nada de per_page=1 pra smoke test.

6. REGRAS DE NEGÓCIO DESCOBERTAS
--------------------------------------------------------

6.1 Status de venda que entram no faturamento da Loja Física:
    REGRA: COMMITTED é o status que representa venda efetivada.
    Portanto:
      INCLUI:  status = COMMITTED (venda aprovada/realizada)
      EXCLUI:  status = PENDING   (aguardando review, não é venda real)
      EXCLUI:  status = CANCELED  (se existir)
    [VER] Confirmar a lista completa de status quando fizermos a
          primeira chamada real a /v1/sales.

6.2 Status de parcela para considerar venda "paga":
    REGRA: olhar payment_status das parcelas (installments/payments).
    Considerar "pago" quando a maioria das parcelas estiver em:
      QUITADO (= RECEBIDO) ou RECEBIDO_PARCIAL
    Vendas com parcelas CANCELADO/PERDIDO entram como perdidas.
    [VER] Decidir com o cliente: vendas parceladas com apenas
          parte das parcelas quitadas contam pro faturamento do
          mês da venda ou do mês do recebimento?
    → Essa decisão vai entrar em DECISOES.txt quando resolvida
      com o Matheus / cliente.

6.3 Tipo de cliente:
    Só clientes INDIVIDUAL (PF) entram nas análises demográficas.
    Clientes ENTITY (PJ) entram no faturamento mas não na
    demografia/gênero/idade/estado (DDD).

6.4 Duplicidade de cliente:
    Cliente novo cadastrado manualmente no caixa pode não ter
    documentNumber. Deduplicar usando (name + phone) como
    fallback — regra a ser afinada na T12.

7. RISCOS TÉCNICOS A CONHECER
--------------------------------------------------------

R1. Migração legada → nova API já deveria ter sido feita até
    2025-11-05. Se a aplicação do cliente Miranda ainda estiver
    usando a API legada, pode haver atrito (ex.: tokens legados
    não funcionam mais). Confirmar com Matheus se a Miranda já
    migrou do lado deles — NÃO é necessariamente automático.

R2. Refresh token single-use exige persistência transacional.
    Se duas cron jobs tentarem renovar o token simultaneamente,
    uma vai falhar. Solução: usar lock no Supabase (advisory lock
    ou coluna token_locked_at com TTL).

R3. Rate limit 600/min × N itens de venda puxados 1 a 1 é o maior
    risco de performance. Investigar na T7 se há endpoint bulk
    de items ou param "expand".

R4. Geo-bloqueio fora do Brasil: Vercel Functions podem rodar em
    região diferente e serem bloqueadas. Duas opções:
      a) Rodar ETL no Supabase Edge Functions (São Paulo).
      b) Configurar Vercel Function com região "gru1" (São Paulo).
    Preferir (a) para ficar colocalizado com o banco.

R5. Sem sandbox. Toda integração testada vai mexer em dados reais
    da conta de teste de 3 dias ou direto na produção do cliente.
    Criar cuidado extra nos endpoints de escrita (não é nosso caso
    imediato — só leitura — mas vale anotar).
