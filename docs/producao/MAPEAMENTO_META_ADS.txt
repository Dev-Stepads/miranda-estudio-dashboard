1. AUTENTICAÇÃO
--------------------------------------------------------
[OK] Tipo: OAuth 2.0 (Facebook Login for Business) + System User Token
     para uso server-to-server de longo prazo.

[OK] CRIAÇÃO DA APP
     1. Ir em https://developers.facebook.com/apps/
     2. Create App → Type: "Business"
     3. Associar a app ao Business Manager da Stepads
        (não ao Business Manager do cliente)
     4. Adicionar produto "Marketing API" ao app
     5. Coletar:
          app_id           — público
          app_secret       — SECRETO (cofre Stepads)

[OK] PERMISSÕES / SCOPES (pedir APENAS o necessário)
     Para leitura de insights de campanhas:
       ads_read               ← ESSENCIAL, é só essa
     Para gerenciar (NÃO é nosso caso, não pedir):
       ads_management
     Para acessar business assets:
       business_management    ← necessário se vamos usar
                                System User token

[OK] FLUXO DE TOKEN — 3 opções, escolher a certa

     (a) User access token (short-lived) — 1-2 horas
         Obtido via OAuth user flow.
         ❌ Inviável para ETL em produção (expira muito rápido).

     (b) Long-lived user access token — 60 dias
         Exchange do short-lived token:
           GET https://graph.facebook.com/v25.0/oauth/access_token
             ?grant_type=fb_exchange_token
             &client_id={app_id}
             &client_secret={app_secret}
             &fb_exchange_token={short_lived_token}
         🟡 Viável mas requer renovação a cada 60 dias.

     (c) 🟢 System User token — NÃO EXPIRA
         Configurado uma vez no Business Manager e não expira
         (a não ser que a gente rotacione, usuário saia do BM,
         ou app seja removida).
         ESTA É A OPÇÃO QUE VAMOS USAR.

[OK] COMO CRIAR O SYSTEM USER TOKEN (passo a passo)

     Precisa do cliente (Miranda) pra algumas etapas:

     1. [Miranda] Abrir Business Manager (business.facebook.com)
     2. [Miranda] Business Settings → Users → System Users
        → Add → "Stepads Dashboard ETL" (nome descritivo)
        → Role: Employee (não precisa Admin)
     3. [Miranda] Assign Ad Accounts ao System User:
        Business Settings → Accounts → Ad Accounts
        → selecionar a ad account da Miranda
        → Add People → System User "Stepads Dashboard ETL"
        → permissão: "View Performance" ou "View Campaigns"
     4. [Miranda ou Stepads] Generate New Token:
        Business Settings → System Users → selecionar o user
        → Generate New Token
        → Selecionar nossa App Marketing
        → Scopes: ads_read, business_management
        → set_token_expires_in_60_days: FALSE (para não expirar)
        → Generate
     5. Copiar o token e GUARDAR IMEDIATAMENTE no cofre Stepads.
        Se perder, precisa gerar um novo (não tem como recuperar).

2. ACCESS TIERS E APP REVIEW
--------------------------------------------------------
[OK] Quando a gente cria uma Marketing API app nova, ela já
     vem com "Standard Access" ao feature "Ads Management
     Standard Access" por DEFAULT.

[OK] Porém, isso é o DEVELOPMENT TIER — limites baixos:
     Rate limit menor, e a app só funciona com ad accounts
     onde quem tem acesso é Admin/dev ou explicitamente
     testador.

[OK] Para o STANDARD TIER (limites maiores, funciona em
     qualquer ad account que o System User tiver acesso),
     é necessário Advanced Access no feature
     "Ads Management Standard Access".

[OK] Advanced Access requer App Review:
     App Dashboard → App Review → Permissions and Features
     → Request Advanced Access em "Ads Management Standard Access"
     → Submeter evidência de uso (screenshot, video do dashboard)

[VER] Nossa aplicação começa em development tier. Na prática,
      **se a Miranda é Admin do BM e a gente cria a app dela
      nesse BM**, pode ser que development tier funcione
      suficientemente para uma conta só. Se os rate limits
      travarem ou se o acesso for bloqueado, subir para
      Advanced. Decidir durante T23-T25 com base no volume
      real de chamadas.

3. RATE LIMITS (BUC — Business Use Case)
--------------------------------------------------------
[OK] Modelo: Business Use Case Rate Limits
     Escopo: por (ad_account × business_use_case)
     Implicação: se endpoint X dispara throttling, endpoint Y
     do MESMO BUC também é bloqueado — quota compartilhada.

[OK] HEADERS RELEVANTES na resposta HTTP:
       X-Business-Use-Case
         Conteúdo JSON com:
           call_count            — chamadas feitas
           total_cputime         — tempo de CPU consumido
           total_time             — tempo total
           estimated_time_to_regain_access — segundos até destravar
       X-Ad-Account-Usage
       X-FB-Ads-Insights-Throttle
       ads_api_access_tier       — indica development vs standard tier

[OK] REGRA DE OURO: o ETL precisa LER esses headers em toda
     chamada e pausar quando call_count estiver alto.
     Implementação recomendada:
       - Se call_count > 80% do permitido → backoff exponencial
       - Se 429 Too Many Requests → esperar estimated_time_to_regain_access
       - Log estruturado de todos os throttles para alertar

[VER] Valores exatos de rate limit não são documentados de
     forma pública (Meta calcula dinâmicamente por BUC).
     Vamos descobrir na prática durante T23-T25.

4. ENDPOINTS QUE VAMOS USAR
--------------------------------------------------------

4.1 INSIGHTS (o principal — tudo do dashboard sai daqui)
    GET https://graph.facebook.com/v25.0/act_{ad_account_id}/insights
    Query params mais importantes:
      access_token={token}
      level=<account|campaign|adset|ad>
      time_range={"since":"2026-04-01","until":"2026-04-10"}
      time_increment=1          ← 1 dia (granularidade diária)
      fields=<lista CSV de fields>
      breakdowns=<CSV opcional>
      action_attribution_windows=['7d_click','1d_view']
      limit=500                  ← paginação
      filtering=[...]            ← opcional

4.2 INSIGHTS POR NÍVEL (vamos chamar todos os 3)
    Campanha:
      GET /act_{id}/insights?level=campaign
    Conjunto de anúncios:
      GET /act_{id}/insights?level=adset
    Anúncio:
      GET /act_{id}/insights?level=ad

    ⚠ ATENÇÃO: NÃO dá pra pedir level=campaign com breakdown
    que implique ad-level (ex.: por criativo). Faça uma chamada
    por level.

4.3 METADADOS (para descobrir o que existe)
    GET /act_{id}/campaigns         — lista campanhas ativas/inativas
    GET /act_{id}/adsets            — lista conjuntos
    GET /act_{id}/ads               — lista anúncios
    GET /act_{id}/adcreatives       — criativos (thumbnail, nome, etc)

    Fields essenciais em cada:
      campaigns: id, name, status, objective, created_time, start_time, stop_time
      adsets:    id, name, status, campaign_id, created_time, targeting
      ads:       id, name, status, adset_id, creative{id,thumbnail_url,image_url,body,title}
      adcreatives: id, name, thumbnail_url, image_url, body, title, call_to_action_type

6. BREAKDOWNS DISPONÍVEIS
--------------------------------------------------------

[OK] Breakdowns demográficos:
       age                     — faixas: 18-24, 25-34, 35-44,
                                 45-54, 55-64, 65+
       gender                  — male, female, unknown
       age,gender              — combinação (mais útil)

[OK] Breakdowns geográficos:
       country
       region                  — estado
       dma                     — designated market area
       impression_device
       device_platform

[OK] Breakdowns de plataforma:
       publisher_platform      — facebook, instagram,
                                 audience_network, messenger
       platform_position       — feed, story, reels, marketplace
       impression_device

[OK] Breakdowns temporais:
       hourly_stats_aggregated_by_advertiser_time_zone
       (não suporta fields de unique_*, reach, frequency)

[OK] LIMITAÇÕES DE BREAKDOWN
     - Só algumas combinações são permitidas (storage constraint)
     - Hourly breakdown → sem reach, sem frequency, sem unique_*
     - reach com age/gender → só últimos 13 meses
     - Alguns breakdowns causam cap ou erro em certos fields

[OK] PARA NOSSO DASHBOARD, vamos usar:
     - Nada de breakdown no endpoint principal (level=campaign|adset|ad)
     - breakdown age,gender APENAS para a aba "Demografia do
       alcance" (se o cliente pedir) — janela de 13 meses
     - breakdown region para mapa de estado
     - breakdown publisher_platform para ver split
       Facebook vs Instagram vs Reels

7. PAGINAÇÃO (cursor-based)
--------------------------------------------------------
[OK] Meta API usa paginação por CURSOR, não offset.

     Resposta típica:
       {
         "data": [...],
         "paging": {
           "cursors": {
             "before": "MTIzNDU2Nzg5MA==",
             "after":  "OTg3NjU0MzIxMA=="
           },
           "next": "https://graph.facebook.com/v25.0/.../?after=OTg3NjU0MzIxMA=="
         }
       }

[OK] Parâmetros:
       limit={N}               — default 25, max 5000
       after={cursor}          — próxima página
       before={cursor}         — página anterior
     Usar limit=500 como padrão (equilíbrio entre throughput
     e tamanho da resposta).

[OK] BEST PRACTICE: quando a query atingir timeout (muito
     comum em insights de contas grandes com janelas longas),
     QUEBRAR em janelas menores de time_range (ex.: 1 dia por
     chamada em vez de 7).

9. WEBHOOKS — NÃO SERVEM PARA INSIGHTS
--------------------------------------------------------
[OK] Meta Marketing API tem webhooks, MAS:
     - Webhooks cobrem EVENTOS (status change, ad rejected,
       spending threshold reached, account restrictions)
     - Webhooks NÃO cobrem atualização de métricas de insights
     - Insights atualizam a cada 15 minutos pelo lado do Meta
       (Ads Reporting API)

[OK] DECISÃO: ETL de insights vai ser 100% POLLING.
     Frequência recomendada: a cada 30 minutos (alinhado com
     o delay natural de 15 min do Meta — não faz sentido
     puxar com mais frequência).

[OK] POSSÍVEL USO FUTURO DE WEBHOOKS:
     Se quisermos notificar quando uma campanha mudar de
     status ou quando um anúncio for rejeitado, webhooks
     podem ser úteis. NÃO é escopo inicial.

10. JANELAS DE COLETA E REPROCESSAMENTO
--------------------------------------------------------
[OK] O Meta RECALCULA dados históricos por alguns dias
     após o dia do evento. Conversões podem "aparecer" até
     7 dias depois. Campanhas podem ser re-otimizadas.

[OK] REGRA DE COLETA DEFENSIVA:
     Toda execução do ETL vai puxar os ÚLTIMOS 7 DIAS
     (não só o dia corrente) e fazer UPSERT no banco.
     Isso garante que qualquer re-cálculo do Meta seja
     refletido automaticamente no nosso dashboard.

[OK] INGESTÃO COMPLETA HISTÓRICA (primeira carga):
     - Dividir em janelas mensais
     - Puxar level=campaign primeiro (menos linhas)
     - Depois level=adset
     - Por último level=ad (maior volume)
     - Respeitando os headers de rate limit

11. CAMPOS DEMOGRÁFICOS DO META — IMPORTANTE
--------------------------------------------------------
[OK] ATENÇÃO: as métricas de age/gender do Meta são do
     PÚBLICO ALCANÇADO/CLIKADO, não dos COMPRADORES.

     Ex: 60% do alcance foi feminino NÃO significa que
     60% das compras foram por mulheres — significa que
     60% das pessoas que VIRAM os anúncios eram mulheres.

[OK] IMPLICAÇÃO PARA O DASHBOARD:
     - Na aba Meta Ads: podemos mostrar "demografia do alcance"
       como um indicador de para QUEM a grana está indo.
     - NUNCA misturar com a demografia de COMPRADORES
       (que vem da Nuvemshop). São coisas diferentes.
     - Labels no dashboard devem deixar isso claro para o
       cliente final não confundir.
