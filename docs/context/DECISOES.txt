[2026-04-10] [ARQUITETURA] Stack do projeto
  Contexto: precisávamos definir hospedagem de backend, banco e frontend
            para um dashboard que consome 3 APIs externas.
  Decisão:  GitHub (repo) + Supabase (banco + funções) + Vercel (deploy do dashboard).
            Contas operacionais: dev@stepads.com.br (Supabase + Vercel).
  Impacto:  Todo o ETL pode rodar em Supabase Edge Functions ou cron jobs.
            Dashboard será uma SPA/Next.js hospedada na Vercel.

[2026-04-10] [NEGOCIO] Escopo da Visão Geral
  Contexto: precisávamos definir o que entra no faturamento consolidado.
  Decisão:  Visão Geral consolida APENAS Loja Física (Conta Azul) +
            E-commerce (Nuvemshop). Meta Ads NÃO entra no faturamento
            consolidado — entra apenas como atribuição na aba própria.
  Impacto:  Evita double-counting. Faturamento atribuído do Meta Ads
            é um recorte informativo, não soma ao total do negócio.

[2026-04-10] [DADOS] Estado do cliente por DDD na Loja Física
  Contexto: Conta Azul pode não trazer estado do cliente direto.
  Decisão:  Derivar estado pelo DDD do telefone quando não houver campo
            de endereço preenchido. Documentar regra de fallback em
            /docs/producao/REGRAS_CONSOLIDACAO.txt.
  Impacto:  Clientes sem telefone ou com DDD inválido entram como
            "Desconhecido" na análise geográfica da Loja Física.

[2026-04-10] [PROCESSO] Revezamento via HANDOFF.txt
  Contexto: projeto será tocado por múltiplas pessoas + Claude em sessões
            distintas, sem memória compartilhada.
  Decisão:  Todo push relevante exige atualização de HANDOFF.txt.
            Antes de qualquer alteração, é obrigatório ler HANDOFF +
            STATUS_ATUAL + mapeamento da fonte envolvida.
  Impacto:  Zero perda de contexto entre sessões. Cria pequeno overhead
            em cada push, mas é intencional.

[2026-04-10] [SEGURANCA] Nunca armazenar credenciais em memória ou em git
  Contexto: credenciais de login (Conta Azul, Nuvemshop) foram
            compartilhadas em chat durante o setup.
  Decisão:  Credenciais ficam apenas em .env.local (gitignored) e em
            gerenciador de senhas. Usuário deve trocar as senhas
            compartilhadas em chat. Para as APIs, usar OAuth/tokens,
            nunca login/senha do painel.
  Impacto:  .gitignore precisa cobrir .env*. Toda nova integração deve

[2026-04-10] [INTEGRACAO] Usar API NOVA do Conta Azul (v1), não a legada
  Contexto: Conta Azul mantém duas APIs em paralelo. A legada
            (api.contaazul.com.br/pub/...) usa ExternalApplicationToken
            + CompanyToken. A nova (api.contaazul.com/v1/...) usa
            OAuth 2.0 padrão e foi lançada em março/2025. O prazo
            oficial de migração para a nova era 2025-11-05, já vencido.
  Decisão:  Todo o desenvolvimento da integração Conta Azul usa
            EXCLUSIVAMENTE a API nova v1. Aplicação registrada em
            https://developers-portal.contaazul.com/ com credenciais
            client_id + client_secret. Fluxo OAuth = authorization_code.
  Impacto:  Refresh token é single-use (2 semanas de validade) — exige
            persistência transacional no Supabase para evitar race
            condition. Access token dura 1 hora. Sem sandbox dedicado:
            testes usam conta trial de 3 dias ou produção do cliente.
            Sem webhooks — sincronização via polling em cron.

[2026-04-10] [INTEGRACAO] ETL do Conta Azul será 100% polling
  Contexto: API nova do Conta Azul não suporta webhooks. Precisamos
            manter os dashboards razoavelmente atualizados ao longo
            do dia sem depender de push do lado da Conta Azul.
  Decisão:  Cron jobs alinhados ao Supabase (ou Vercel Cron com região
            gru1) executando:
              - ingestão incremental a cada 30 min em horário comercial
              - ingestão completa do dia anterior uma vez por dia (02h)
            Preferência: rodar no Supabase Edge Functions (colocalizado
            com o banco em São Paulo) para evitar possível geo-bloqueio
            da API Conta Azul fora do Brasil.
  Impacto:  Rate limit 600 req/min × N chamadas /sales/{id}/items por
            venda (endpoint separado) é o maior risco de performance.
            Na T7 validar se existe param "expand=items" que permita
            trazer items inline em /sales — impacto direto no desenho
            do loop de ingestão.

[2026-04-10] [INTEGRACAO] Separar canal de venda por "vendedor_responsavel"
  Contexto: Investigação no dashboard real do Conta Azul da Miranda
            (venda #16138, 2026-04-10) mostrou um campo "Vendedor
            responsável" com valor "LOJA FÍSICA". Esse campo é usado
            como etiqueta de canal pela Miranda — todas as vendas

            produtos" lista todas as vendas (incluindo PDV/frente de

  Decisão:  O ETL do Conta Azul vai filtrar vendas da loja física

            (ou valores equivalentes a confirmar na T7/T8).
  Impacto:  A query canônica para sales da Conta Azul vai ter um filtro
            adicional por vendedor. Precisamos mapear na T7 a LISTA
            COMPLETA de valores usados pela Miranda nesse campo — se
            houver variações tipo "LOJA FÍSICA - CAIXA 1", ou
            vendedores individuais para comissão, o filtro precisa
            acomodar todos. Em caso de dúvida, puxar TUDO e filtrar
            no ETL (mais seguro que depender de query filter externo).

[2026-04-10] [INTEGRACAO] Nuvemshop — webhooks primários + polling safety-net
  Contexto: A API Nuvemshop suporta webhooks (order/paid, order/cancelled,
            customer/created, etc.) com política de retry de 48h. Webhook
            sozinho pode perder eventos se nosso endpoint ficar fora do ar
            por mais de 48h. Polling sozinho desperdiça rate limit e cria
            latência desnecessária.
  Decisão:  Híbrido. Webhook como caminho primário para eventos de order
            e customer. Polling defensivo:
              - /orders com janela das últimas 48h (1x/dia, reconciliação)
              - /checkouts a cada 6h (carrinhos abandonados — retenção
                de 30 dias exige frequência mínima)
            Webhooks chegam via endpoint
              https://miranda-estudio-dashboard.vercel.app/api/webhooks/nuvemshop
            Validação HMAC-SHA256 obrigatória usando o client_secret
            como chave. Processamento é assíncrono (responde 200 em
            menos de 3s, enfileira o resource_id para um worker).
  Impacto:  Arquitetura do ETL da Nuvemshop é diferente da do Conta Azul
            (que é 100% polling por falta de webhooks). Código dos dois
            não deve ser reusado direto — mas a normalização final para
            tabelas canônicas (sales, customers) é a mesma.

[2026-04-10] [INTEGRACAO] Meta Ads: ETL 100% polling a cada 30 min
  Contexto: Marketing API tem webhooks, mas eles só cobrem EVENTOS
            (status change, ad rejected, account issues). Para
            métricas de insights (spend, impressões, conversões),
            webhooks NÃO são uma opção. A Ads Reporting API
            atualiza os dados do lado do Meta a cada 15 minutos.
  Decisão:  ETL polling em /insights:
              - Frequência: a cada 30 min
              - Níveis puxados: campaign, adset, ad (3 chamadas)

            Não usamos webhooks do Meta no escopo inicial.
  Impacto:  Dashboard tem latência de até 30-45 min em relação ao
            Ads Manager (15 min do Meta + 30 min do nosso cron).
            Aceitável para todos os KPIs que o cliente pediu.

[2026-04-11] [DADOS] Convenções do schema inicial do Supabase
  Contexto: primeira migration cria as tabelas raw_* (staging) e
            canônicas (sales, sale_items, products, customers,
            abandoned_checkouts, meta_ads_insights). Várias escolhas
            de tipo e convenção afetam tudo que vem depois.
  Decisão:  1) PKs são bigserial (bigint auto-incrementado), NÃO
               uuid. É mais denso, mais rápido pra joins, e o
               workload não precisa de IDs imprevisíveis publicamente.
               IDs de fonte externa (source_sale_id etc.) ficam em
               coluna separada tipo text.
            2) Enums são text + CHECK constraint, NÃO tipo ENUM
               nativo do PG. Motivo: adicionar valor num ENUM nativo
               exige ALTER TYPE, enquanto CHECK é alterado em uma
               migration comum. As APIs que consumimos podem
               introduzir status novo a qualquer momento.
            3) Raw tables guardam o payload completo da API em uma
               coluna jsonb + algumas colunas pulled-out (source_id,
               date) pra indexar. Append-only. Isso permite reprocessar
               o canônico a partir do histórico sem precisar rechamar
               as APIs.
            4) Todos os timestamps são timestamptz (UTC on disk).
               Conversão pra America/Sao_Paulo acontece na view ou
               aplicação. Datas puras (Meta insights diários) usam date.
            5) Valores monetários são numeric(14,2). Suficiente pra
               999 bi com centavo — overkill, mas sem custo.
            6) RLS habilitado em TODAS as tabelas desde o início,
               sem policies. Resultado: anon/authenticated são negados
               por padrão; ETL usa service_role que tem bypass. Quando
               o frontend começar a ler dados, policies de leitura
               entram em views dedicadas, nunca nas tabelas raw.
            7) SKU em products é único só quando não-nulo (partial
               unique index). Produtos sem SKU podem coexistir, o
               que é comum no catálogo da Miranda.
            8) meta_ads_insights tem unique (date, level, campaign_id,
               coalesce(adset_id,''), coalesce(ad_id,'')) — truque
               pra unique com colunas nullable.
  Impacto:  Define o "jeito Miranda" de modelar. Qualquer tabela
            nova deve seguir esse padrão (RLS on, timestamptz, bigserial,
            numeric pra dinheiro, CHECK pra enum). Migrations futuras
            podem referenciar esta decisão em vez de reargumentar.

[2026-04-11] [INTEGRACAO] Conta Azul — autenticação é AWS Cognito, não OAuth com scopes granulares
  Contexto: durante o registro da app de produção no portal
            developers-portal.contaazul.com (sessão 2026-04-11),
            descobrimos que o que o Conta Azul chama de "OAuth 2.0" é,
            na prática, um wrapper sobre AWS Cognito (issuer
            cognito-idp.sa-east-1.amazonaws.com/sa-east-1_Vp83J11wA,
            confirmado decodificando o JWT do access_token).
            Os scopes não são "sales:read", "customers:read" como o
            MAPEAMENTO_CONTA_AZUL.txt §1 supôs ([VER]). Os REAIS são:
              openid profile aws.cognito.signin.user.admin
            Não há tela de seleção de scopes no portal — é fixo.
  Decisão:  - Aceitar que a autorização granular vem do PERFIL DO ERP
              USER que loga no fluxo OAuth, não dos scopes. Ou seja,
              o que a app pode ler depende das permissões do usuário
              da Miranda que autorizar a app, não do que a gente pediu.
            - Atualizar MAPEAMENTO_CONTA_AZUL.txt §1 (passar de [VER]
              pra [OK] com os scopes Cognito reais).
            - No código (T7), passar exatamente esses 3 scopes na URL
              de authorize: openid+profile+aws.cognito.signin.user.admin
  Impacto:  Não dá pra restringir privilégios da app via scopes. A
            única "defesa" é orientar a Miranda a autorizar com um
            usuário ERP que tenha só permissão de leitura. Documentar
            isso no checklist de onboarding do cliente quando a app
            for pra produção real.

[2026-04-11] [INTEGRACAO] Conta Azul — estratégia DEV app + PROD app (1 redirect por app)
  Contexto: o portal do Conta Azul permite apenas UM redirect_uri
            por aplicação registrada (verificado visualmente em
            2026-04-11). Não há campo "+ adicionar outra URL".
            A gente precisa de redirects diferentes pra
            desenvolvimento local (localhost) e produção (Vercel).
  Decisão:  Usar DUAS aplicações distintas:
            (1) App DEV — auto-criada pelo portal quando você se
                cadastra. Vem com redirect_uri https://contaazul.com
                (placeholder pra dev copiar o ?code= manualmente da
                barra de endereço) e um ERP user de teste com trial
                de ~3 dias. Credenciais salvas como
                CONTA_AZUL_DEV_* no .env.local. Usada para testar
                o fluxo OAuth durante o desenvolvimento do código.
            (2) App PROD — "Miranda Dashboard" criada manualmente
                em 2026-04-11. CNPJ Stepads (32.972.296/0001-07).
                redirect_uri =
                  https://miranda-estudio-dashboard.vercel.app/api/auth/contaazul/callback
                Credenciais salvas como CONTA_AZUL_* (sem _DEV).
                Usada quando a Miranda autorizar de verdade (T7+).
  Impacto:  Código do client OAuth Conta Azul deve ler as variáveis
            CONTA_AZUL_DEV_* em modo dev e CONTA_AZUL_* em prod
            (controlado por NODE_ENV ou flag explícita). Mappers,
            HTTP base e tipos são iguais — só client_id, secret e
            redirect_uri trocam. Cliente secret do Conta Azul é
            visível livremente ao reabrir a app no portal (não é
            one-shot), o que reduz o stress operacional.

[2026-04-11] [DADOS] Views do dashboard — granularidade e convenções
  Contexto: precisávamos decidir o shape das 14 views que alimentam
            as 4 abas do dashboard. Duas opções principais: views
            single-row com KPIs agregados (uma view por card) ou
            views granulares (diário/por entidade) com frontend
            agregando. Também: como lidar com filtro de período,
            timezone e segurança.
  Decisão:  1) GRANULARIDADE DIÁRIA OU POR ENTIDADE.
               Todas as views retornam linhas granulares (por dia,
               por produto, por estado, por gênero, etc.), NUNCA
               single-row KPIs. Razão: PostgreSQL views não aceitam
               parâmetros — se a view agregasse "total do período",
               o período teria que ser hardcoded ou virar function.
               Com granular + WHERE no frontend, o mesmo dataset
               serve "hoje", "últimos 7d", "mês atual" etc. PostgREST
               (camada Supabase) faz o pushdown automático.
            2) KPIs SINGLE-VALUE SÃO CALCULADOS NO FRONTEND.
               total_revenue, avg_ticket, share_canal_pct, ROAS
               consolidado, etc., são derivados das agregações
               diárias via SUM/AVG no client. Menos views, mais
               flexibilidade de filtro.
            3) TIMEZONE APLICADO NA VIEW.
               `(sale_date at time zone 'America/Sao_Paulo')::date`
               transforma timestamptz em date no fuso correto. O
               frontend consome só `day date` e nunca precisa se
               preocupar com conversão.
            4) FILTRO DE REVENUE = status='paid' APENAS.
               'cancelled', 'refunded', 'pending' ficam de fora.
               'partially_paid' da Nuvemshop já é mapeado pra 'paid'
               na canônica (ver REGRAS §2.2). Se o representante
               definir diferente, o filtro muda SÓ nas views — ETL
               e schema ficam iguais.
            5) META ADS: apenas level='campaign' no v_meta_campanha_daily.
               Adset e ad têm views de ranking próprias. Misturar níveis
               causaria double-counting (adset já soma em campaign).
            6) VISÃO GERAL NÃO INCLUI META.

[2026-04-13] [DADOS] Meta Ads unique constraint: drop expression
             index, trocar por constraint real em colunas simples
  Contexto: O schema original (migration 20260411120000) criou um
            unique index com expressao:
              create unique index uq_meta_insights on meta_ads_insights (
                date, level, campaign_id, coalesce(adset_id,''), coalesce(ad_id,'')
              );
            A intencao era tratar adset_id/ad_id NULL (em rows do
            nivel campaign, por exemplo) como tuplas unicas. Mas o
            Supabase JS client `.upsert({ onConflict: '...' })` so
            funciona com UNIQUE CONSTRAINT real em colunas SIMPLES —
            expression indexes nao sao alcancaveis via PostgREST
            `on_conflict`. Isso quebra o upsert do ETL.
  Decisao:  Migration 20260413120000_meta_ads_account_view.sql
            transforma adset_id/ad_id em NOT NULL DEFAULT '' e
            substitui o expression index por UNIQUE CONSTRAINT real:
              UNIQUE (date, level, campaign_id, adset_id, ad_id)
            Semantica preservada: '' = "nao aplicavel pro nivel"
            (campaign rows tem adset_id='' e ad_id=''). Views
            existentes ja filtram por level, entao '' nao aparece
            nos rankings.
  Impacto:  Upsert do ETL Meta Ads funciona via Supabase JS. O
            mapper continua produzindo null no canonical type (boa
            semantica), mas o sync converte pra '' no boundary do
            upsert. Futuro: se alguma feature precisar de "rows
            tipicamente sem adset_id", verificar se esse default
            atrapalha.
